Linux Fundamentals Exercises (Hands-on)

Designed for beginners → intermediate learners. Works on most Linux distros (Ubuntu/Debian, Fedora/RHEL, Arch). Commands are safe-by-default and run inside a practice directory.

How to use this:
  1. Open a terminal on a Linux machine (VM, cloud shell, WSL, or native).
  2. Do exercises in order (or jump by topic).
  3. Use <details> sections only after trying yourself.

Tip: If a command behaves differently on your distro, that’s normal—Linux is a family of systems.

0) Setup

Exercise 0.1 — Create a safe lab folder

Goal: Create a dedicated practice directory so you don’t touch important system files.

  1. Create the directory ~/linux-fundamentals-lab.
  2. Inside it, create subfolders: data, notes, bin.
  3. Print the absolute path to confirm where you are.
  • I created the folder tree.
  • I can see the path with pwd.
Show solution 
mkdir -p ~/linux-fundamentals-lab/{data,notes,bin}
cd ~/linux-fundamentals-lab
pwd
ls -la

Expected: you see data, notes, bin directories.

Exercise 0.2 — Identify your shell and distro

Goal: Learn what environment you’re in (useful for troubleshooting).

  1. Print your current shell.
  2. Print your kernel version.
  3. Print OS information (distro name/version).

Exercise 2.4 — Hard links vs symlinks (basic)

Goal: Understand links by observing behavior.

  1. Create a file original.txt with a short line of text.
  2. Create a hard link hardlink.txt pointing to it.
  3. Create a symlink symlink.txt pointing to it.
  4. Show link count with ls -li.
  5. Delete original.txt. Observe what happens when you cat the link files.
Safety: Do this only inside the lab folder.
Show solution 
cd ~/linux-fundamentals-lab/data
printf "hello links\n" > original.txt
ln original.txt hardlink.txt
ln -s original.txt symlink.txt

ls -li original.txt hardlink.txt symlink.txt
rm original.txt

# Hard link still has the content; symlink breaks
cat hardlink.txt
cat symlink.txt

Expected: hard link keeps working; symlink reports “No such file or directory”.

4) Redirection & Pipes

Exercise 4.1 — stdout vs stderr

Goal: Understand standard output and standard error.

  1. Run a command that succeeds and one that fails.
  2. Redirect stdout to a file.
  3. Redirect stderr to a file.
  4. Redirect both stdout and stderr to the same file.
Show solution 
cd ~/linux-fundamentals-lab

# Success (stdout)
ls /etc > out.txt

# Failure (stderr) - file likely doesn't exist
ls /this/does/not/exist 2> err.txt

# Both together
( ls /etc; ls /this/does/not/exist ) > all.txt 2>&1

# Inspect
wc -l out.txt err.txt all.txt
tail -n 5 all.txt

Exercise 4.2 — Pipes: count and sort

Goal: Use pipelines to transform data.

  1. List files in your lab data folder.
  2. Count how many entries there are.
  3. Sort by name and display one per line.
Show solution 
cd ~/linux-fundamentals-lab/data
ls | wc -l
ls -1 | sort

Exercise 4.3 — Build a simple CSV report

Goal: Generate a small report from command output.

  1. Create a file report.csv in your lab root with a header: name,size_bytes.
  2. Append lines for each file in data showing filename and size in bytes.
Show solution 
cd ~/linux-fundamentals-lab
printf "name,size_bytes\n" > report.csv

# GNU stat (common on Linux):
for f in data/*; do
  [ -f "$f" ] || continue
  printf "%s,%s\n" "$(basename "$f")" "$(stat -c %s "$f")" >> report.csv
done

column -s, -t report.csv | head

If stat -c doesn’t work (rare on Linux), your system might be using BSD stat syntax.

5) Finding Things (find/grep)

Exercise 5.1 — Find files by name

Goal: Use find to locate files under a directory tree.

  1. Find all .txt files under ~/linux-fundamentals-lab.
  2. Find files whose name contains “log”.

find ~/linux-fundamentals-lab -type f -name "*.txt"
find ~/linux-fundamentals-lab -type f -name "*log*"

Exercise 5.2 — Grep for content

Goal: Search inside files for text patterns.

  1. Add the word ERROR on one line in any file under ~/linux-fundamentals-lab.
  2. Search recursively for ERROR and show matching line numbers.
  3. Show only filenames that contain matches.
Show solution 
cd ~/linux-fundamentals-lab
printf "ERROR: something happened\n" >> notes/about.txt

grep -RIn "ERROR" .
grep -RIl "ERROR" .

Exercise 5.3 — Combine find + grep safely

Goal: Search only within a filtered file set.

  1. Search for the word Goal only in .txt files under notes.
  2. Do it using find ... -exec (no pipes required).
Show solution 
cd ~/linux-fundamentals-lab
find notes -type f -name "*.txt" -exec grep -nH "Goal" {} \;

Flags meaning: -n line numbers, -H show filename.

6) Permissions

Exercise 6.1 — Read permissions from ls -l

Goal: Decode rwx for user/group/others.

  1. In your lab folder, run ls -l on a file.
  2. Write down what the first 10 characters mean (e.g., -rw-r--r--).
  3. Identify: file type, user perms, group perms, other perms.
Show solution

Example: -rw-r--r--

  • - = regular file (could also be d directory, l symlink)
  • rw- = user can read/write
  • r-- = group can read
  • r-- = others can read

Exercise 6.2 — chmod with symbolic modes

Goal: Modify permissions using u/g/o and +/-/=.

  1. Create script.sh in ~/linux-fundamentals-lab/bin with a simple echo.
  2. Make it executable for your user only.
  3. Run it using ./script.sh.
  4. Remove execute permission again.
Show solution 
cd ~/linux-fundamentals-lab/bin
cat > script.sh <<'EOF'
#!/usr/bin/env bash
echo "Hello from script"
EOF

# user execute only; remove exec from group/others
chmod u+x,go-x script.sh

./script.sh

chmod u-x script.sh

Exercise 6.3 — chmod with numeric (octal) modes

Goal: Understand chmod 600/644/755 patterns.

  1. Create a file secret.txt in notes.
  2. Set permissions to: user read/write, nobody else any access.
  3. Verify using ls -l.
  4. Change to: user read/write, group read, others read.
Show solution 
cd ~/linux-fundamentals-lab/notes
printf "top secret\n" > secret.txt

chmod 600 secret.txt
ls -l secret.txt

chmod 644 secret.txt
ls -l secret.txt

Octal: r=4, w=2, x=1. So 6=rw-, 4=r--.

Exercise 6.4 — Default permissions (umask)

Goal: Observe how umask impacts new files/dirs.

  1. Check your current umask.
  2. Create a new file and a new directory.
  3. Check their permissions with ls -l and ls -ld.
Show solution 
cd ~/linux-fundamentals-lab
umask
touch umask_file.txt
mkdir -p umask_dir
ls -l umask_file.txt
ls -ld umask_dir

Typical default: files become 644, dirs become 755 (but this can vary).

Goal: Use job control: &, jobs, fg, bg.

  1. Start a command that runs ~30 seconds in the background.
  2. List jobs.
  3. Bring it to foreground.
  4. Stop it (Ctrl+Z), then resume in background.
Show solution

Exercise 7.3 — Signals: terminate a process you started

Goal: Practice kill safely on your own process.

  1. Start a long sleep in background.
  2. Find its PID.
  3. Send SIGTERM, confirm it ended.
  4. If still running, send SIGKILL (last resort).
Show solution 
sleep 9999 &
ps -o pid,cmd -u "$USER" | grep "sleep 9999" | grep -v grep

# Suppose PID is 12345:
kill 12345
ps -p 12345 || echo "Process ended"

# Last resort:
kill -9 12345

Exercise 9.1 — Identify your package manager

Goal: Figure out whether you’re on apt, dnf, yum, pacman, etc.

  1. Check which of these exist: apt, dnf, yum, pacman, zypper.
  2. Pick the one that exists and find a package (e.g., curl) without installing it.
Show solution

command -v apt dnf yum pacman zypper | cat

# Fedora/RHEL example:
dnf info curl

# Arch example:
pacman -Si curl

Exercise 9.2 — Check service status (systemd)

Goal: Learn the basics of systemctl (common on modern distros).

  1. Check whether systemd is present.
  2. List running services.
  3. Check status of a common service (try ssh or sshd).
Show solution 
# Is systemd present?
ps -p 1 -o comm=

# List running services (systemd)
systemctl list-units --type=service --state=running

# Check ssh service name (varies by distro)
systemctl status ssh || systemctl status sshd

If your system isn’t using systemd (rare), these commands may not work.

Exercise 9.3 — View logs with journalctl

Goal: Read system logs without guesswork.

  1. View the most recent 50 log lines.
  2. View logs for a specific service (ssh/sshd).
  3. Follow logs live for ~10 seconds, then stop.
Show solution 
journalctl -n 50

# Service logs:
journalctl -u ssh -n 50 || journalctl -u sshd -n 50

# Follow live (Ctrl+C to stop)
journalctl -f

10) Mini Project — Build a “system snapshot” script

Exercise 10.1 — Create a snapshot script and save output

Goal: Combine several fundamentals into a practical script.

Requirements: Create ~/linux-fundamentals-lab/bin/snapshot.sh that:

  1. Prints the current date/time.
  2. Prints hostname and kernel info.
  3. Prints uptime.
  4. Prints disk usage for your home filesystem.
  5. Prints top 5 memory-consuming processes (best-effort).
  6. Saves all output to ~/linux-fundamentals-lab/notes/snapshot.txt.
  • Script runs without errors.
  • Output is written to snapshot.txt.
  • Script is executable.
Show one possible solution 
cd ~/linux-fundamentals-lab

cat > bin/snapshot.sh <<'EOF'
#!/usr/bin/env bash
set -euo pipefail

OUT="$HOME/linux-fundamentals-lab/notes/snapshot.txt"

{
  echo "=== SNAPSHOT ==="
  echo "Time: $(date -Is)"
  echo
  echo "Host: $(hostname)"
  echo "Kernel: $(uname -r)"
  echo
  echo "Uptime:"
  uptime || true
  echo
  echo "Disk usage (home filesystem):"
  df -h "$HOME" || true
  echo
  echo "Top 5 processes by RSS (memory):"
  # RSS is in KB; sort descending; show first 5 processes after header
  ps -eo pid,comm,rss --sort=-rss | head -n 6 || true
  echo
  echo "Network (IP addresses):"
  ip -brief addr || ip addr || true
  echo "=== END ==="
} | tee "$OUT"
EOF

chmod u+x bin/snapshot.sh
./bin/snapshot.sh
ls -l notes/snapshot.txt
tail -n 20 notes/snapshot.txt

Notes:
• set -euo pipefail makes scripts safer (fail fast).
• Some commands may differ slightly by distro; the script uses “best-effort” fallbacks.

Next steps (optional):
  • Add argument support: snapshot.sh --out /path/file
  • Write snapshots to a timestamped filename
  • Compress output using gzip or tar
